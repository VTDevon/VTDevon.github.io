Test Procedure for Build: Phase 2

Purpose of the Build

The primary goal of Phase 2 is to validate the middleware and load balancers' communication with backend components and ensure reliable handling of traffic and user notifications. This phase bridges the backend with user-facing systems, making it crucial to test performance, scalability, security, and notification mechanisms.

Components Included

  • From Phase 1:
    • User authentication and management service.
    • Fitness logging and nutrition tracking service.
    • Database (AWS RDS).
    • Third-party API integration.
    • Analytics and progress computation module.
  • New Components:
    • Load balancers to distribute traffic efficiently between frontend and backend.
    • Notification service for sending user alerts (email and push notifications).

Key Testing Objectives

  • Load Balancer Communication: Validate that load balancers correctly route traffic to backend services and ensure balanced distribution of requests under normal and high traffic conditions.
  • Performance and Scalability: Simulate user loads to measure response times and detect bottlenecks. Validate system performance under peak loads.
  • Notification Service: Test notification triggers for workout reminders and nutrition updates. Verify proper formatting and delivery of notifications (both email and push).
  • Middleware Security: Test for secure communication between the middleware and backend using HTTPS. Simulate malicious traffic (e.g., DDoS attacks) to test the load balancer's resilience.
  • Data Integrity and Synchronization: Verify that data transmitted through the middleware matches the backend database records.

Test Case Skeletons

  • Load Balancer Routing Validation:
    • Setup: Configure the load balancers with backend service endpoints.
    • Actions:
      1. Send REST API requests to the system via the load balancer.
      2. Log the routing of each request to ensure it reaches the intended backend service.
    • Expected Outcome: Each request is routed to the correct service without errors or significant delay.
  • Scalability Testing:
    • Setup: Use a load-testing tool (e.g., JMeter) to simulate increasing numbers of concurrent users.
    • Actions:
      1. Start with a baseline of 100 users, gradually increasing to 10,000 concurrent users.
      2. Measure response times, throughput, and error rates.
    • Expected Outcome: The system maintains response times under a specified threshold (e.g., <200ms) without dropping requests.
  • Notification Trigger Validation:
    • Setup: Configure sample notification templates for email and push notifications.
    • Actions:
      1. Simulate scenarios (e.g., logging a missed workout, updating nutrition goals).
      2. Observe if notifications are triggered and sent.
    • Expected Outcome: Notifications are delivered promptly with accurate content.

Required Overhead Software

  • Testing Tools: Apache JMeter, Locust, or equivalent; Postman for API testing; OWASP ZAP for security validation.
  • Stubs and Mock Services: Mock frontend to simulate user requests; stubbed third-party API responses for controlled testing.
  • Logging and Monitoring: AWS CloudWatch or equivalent for performance monitoring during tests.
