Phase 1: Backend and Microservices Test Procedures

Overview

Phase 1 focuses on testing the core backend systems and microservices that form the foundation of the Health and Fitness Tracking Application. The goal is to ensure the backend services function as intended and can handle user data securely and efficiently. By validating the functionality and integration of microservices, database interactions, and third-party APIs in this phase, we lay the groundwork for reliable middleware and frontend integration in later phases.

Components Included

  • User Authentication and Management Service: Handles user login, registration, and profile management, ensuring secure access and data integrity.
  • Fitness Logging and Nutrition Tracking Service: Allows users to log workouts and nutrition data, storing this information in the database for later analysis.
  • Database (AWS RDS): Stores user profiles, fitness logs, nutrition entries, and other application data.
  • Third-Party API Integration: Fetches nutrition and workout data from external APIs to enrich the user experience.
  • Analytics and Progress Computation Module: Computes fitness metrics, aggregates user data, and generates progress reports and insights.

Main Purpose of Phase 1 Testing

The main objective of Phase 1 is to validate that each backend component operates independently and interacts correctly with other components. The focus includes verifying REST API functionality, database schema and operations, integration with third-party APIs, and the accuracy of analytics calculations. This ensures a robust backend before introducing additional layers such as middleware and frontend interfaces.

Test Environment

  • Backend Deployment: Hosted on a staging environment replicating the production setup, including AWS RDS for database storage and mock services for third-party APIs.
  • API Testing Tools: Postman or similar tools for testing REST API endpoints.
  • Database Management Tools: Tools like MySQL Workbench for validating database schema and CRUD operations.
  • Mock API Services: Simulates responses from third-party APIs to test integrations without external dependencies.

Test Procedure and Case Skeletons

Below is a list of test case skeletons designed to validate the functionality of individual backend components and their interactions. Specific test data and expected results will be refined in later stages.

User Authentication and Management Service

  • TC-BE-01: Test user registration with valid data. Verify the user is stored in the database, and credentials are securely hashed.
  • TC-BE-02: Test user login with valid and invalid credentials. Ensure valid credentials return a session token, and invalid credentials result in appropriate error messages.
  • TC-BE-03: Test profile updates (e.g., changing goals). Verify updates are reflected in the database and are accessible through API calls.

Fitness Logging and Nutrition Tracking Service

  • TC-BE-04: Test logging a workout (exercise type, duration, intensity). Verify the data is stored correctly and retrieved for progress analysis.
  • TC-BE-05: Test logging nutrition data (meals, calories, macronutrients). Confirm that entries are saved and linked to the correct user profile.
  • TC-BE-06: Test invalid input for workouts or nutrition (e.g., negative calorie values). Ensure the system rejects invalid data and provides meaningful error messages.

Database Operations

  • TC-BE-07: Verify the database schema against requirements. Confirm all tables and fields align with the expected design.
  • TC-BE-08: Test CRUD operations (Create, Read, Update, Delete) for user profiles, workouts, and nutrition entries. Ensure all operations maintain data integrity.
  • TC-BE-09: Test data retrieval for progress analysis (e.g., fetching workouts for a specific date range). Validate performance and accuracy of results.

Overhead Software (Drivers, Stubs, and Test Tools)

  • API Test Automation: Use tools like Postman or JMeter to automate testing of REST API endpoints.
  • Mock API Services: Simulate third-party APIs to test system responses under controlled conditions.
  • Database Testing Frameworks: Use tools like Flyway for schema validation and data integrity tests.
  • Error Simulation Drivers: Inject artificial errors (e.g., API timeouts, database connection failures) to test system resilience.
  • Performance Testing Tools: Use tools like Apache JMeter to test scalability and response times under simulated load.

Rationale for Phase 1 Strategy

Focusing on the backend and microservices in Phase 1 ensures the foundation of the system is reliable before introducing middleware and frontend layers. By validating core services, database operations, and external API integrations early, we can address potential issues at their source, reducing the risk of cascading failures during later integration phases. This approach also minimizes reliance on stubs for backend functionality in subsequent phases, enabling more realistic end-to-end testing.
